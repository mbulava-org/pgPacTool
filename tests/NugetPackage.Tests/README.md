# NuGet Package Validation Tests

This test project ensures that the NuGet packages generated by pgPacTool are properly constructed with all required dependencies and can be consumed successfully.

## Purpose

After experiencing missing reference issues with `pgpac` and `mbulava.PostgreSql.Dac` packages, these tests were created to:

1. **Validate Package Structure** - Ensure all required files are included in the packages
2. **Verify Dependencies** - Confirm all dependencies are correctly specified or embedded
3. **Test Consumption** - Verify packages can be installed and used in real projects
4. **Check Native Libraries** - Ensure native libpg_query libraries for all platforms are included
5. **Test Global Tool** - Verify the pgpac global tool can be installed and executed

## Tests

### Package Validation Tests

#### 1. DacLibraryPackage_ShouldContainAllRequiredFiles
Validates the structure of `mbulava.PostgreSql.Dac` package:
- Main library DLL is present
- Npgquery.dll is embedded (not a NuGet dependency)
- Native libraries for Windows, Linux, and macOS are included
- Multiple PostgreSQL versions (16, 17) are supported
- Correct NuGet dependencies (Google.Protobuf, Npgsql)

### 2. GlobalToolPackage_ShouldContainAllRequiredFiles
Validates the structure of `postgresPacTools` global tool package:
- Tool executable and all dependencies are in tools directory
- Native libraries are included in the tool package
- DotnetTool package type is correctly set
- Tool command name is correctly configured

### 3. DacLibraryPackage_CanBeConsumedSuccessfully
Tests that the `mbulava.PostgreSql.Dac` package can be:
- Added to a new .NET 10 console project
- Restored successfully
- Compiled successfully
- Executed successfully

### 4. GlobalTool_CanBeInstalledAndExecuted
Tests that the `postgresPacTools` global tool can be:
- Installed from a local NuGet feed
- Executed with --version flag
- Executed with --help flag

### 5. DacLibraryPackage_NativeLibrariesLoadCorrectly
Tests that native libraries load correctly when consuming the package:
- Creates a project that uses Npgquery parser
- Builds and runs the project
- Verifies native libraries (libpg_query) are loaded successfully
- Tests actual SQL parsing functionality

### Pagila Integration Tests

These tests validate the complete MSBuild.Sdk.PostgreSql workflow using the Pagila sample database schema in a real PostgreSQL database running in a Docker container.

**⚠️ REQUIRES DOCKER**: These tests **require Docker to be running** and will **fail intentionally** if Docker is not available. This ensures CI/CD pipelines properly configure Docker support.

#### 6. PagilaDatabase_CanBeExtractedToMSBuildProject
Tests extraction from live database to MSBuild SDK-style project:
- Deploys Pagila schema to PostgreSQL
- Extracts schema to `.csproj` file using `MSBuild.Sdk.PostgreSql` SDK
- Verifies project structure contains correct SDK reference and DatabaseName
- Tests the **proper** extraction workflow (not JSON files)

#### 7. PagilaDatabase_MSBuildProject_CanBeCompiled
Tests the MSBuild SDK compilation workflow:
- Extracts database to `.csproj` format
- Compiles the project using `pgpac compile` command
- Verifies `.pgpac` output file is created
- Tests dependency validation and circular reference detection

#### 8. PagilaDatabase_ExtractCompilePublish_RoundTripWorks
Tests the complete round-trip workflow:
- **Extract**: Live database → `.csproj` MSBuild project
- **Compile**: `.csproj` → `.pgpac` package file
- **Publish**: `.pgpac` → New database deployment
- Verifies schema consistency (table count matches)
- Tests the **recommended production workflow**

**Key Points**:
- Uses **MSBuild.Sdk.PostgreSql** projects (`.csproj`), not raw JSON files
- Tests **`.pgpac`** package format (the compiled output)
- JSON files (`.pgproj.json`) are considered obsolete/temporary
- Validates the proper tooling workflow that users should follow

## Running the Tests

```powershell
# Run all tests
dotnet test tests/NugetPackage.Tests/NugetPackage.Tests.csproj

# Run only package validation tests (no Docker required)
dotnet test tests/NugetPackage.Tests/NugetPackage.Tests.csproj --filter "FullyQualifiedName~NugetPackageValidationTests"

# Run only Pagila integration tests (Docker required)
dotnet test tests/NugetPackage.Tests/NugetPackage.Tests.csproj --filter "FullyQualifiedName~PagilaIntegrationTests"

# Run specific test
dotnet test tests/NugetPackage.Tests/NugetPackage.Tests.csproj --filter "FullyQualifiedName~DacLibraryPackage_ShouldContainAllRequiredFiles"

# Run with verbose output
dotnet test tests/NugetPackage.Tests/NugetPackage.Tests.csproj --logger "console;verbosity=detailed"
```

## Docker Requirements

**Pagila Integration Tests**:
- Require Docker Desktop or Docker Engine to be running
- Use PostgreSQL 16 container (automatically pulled if not present)
- Will **fail intentionally** if Docker is not available
- This ensures CI/CD pipelines properly support Docker

**Package Validation Tests**:
- Do not require Docker
- Can run in any environment with .NET 10 SDK

## How It Works

The tests use a comprehensive approach:

1. **Build Packages**: Builds both `mbulava.PostgreSql.Dac` and `postgresPacTools` packages in Release mode
2. **Create Local Feed**: Creates a temporary local NuGet feed for testing
3. **Validate Structure**: Uses NuGet.Packaging APIs to inspect package contents
4. **Test Consumption**: Creates temporary test projects that consume the packages
5. **Verify Functionality**: Runs the test projects to ensure everything works
6. **Clean Up**: Removes temporary files after tests complete

## Key Features

- **Isolated Testing**: Each test uses a unique temporary directory
- **No Side Effects**: Tests clean up after themselves
- **Comprehensive Validation**: Tests both structure and functionality
- **Cross-Platform**: Validates native libraries for Windows, Linux, and macOS
- **Version Testing**: Ensures multiple PostgreSQL versions (16, 17) are supported

## Adding New Tests

To add a new test:

1. Add a new `[Fact]` method to `NugetPackageValidationTests.cs`
2. Use the helper methods:
   - `BuildPackage(projectName)` - Builds a package
   - `PublishPackageToLocalFeed(packagePath)` - Publishes to local feed
   - `CreateTestProject(dir, package, includeTestCode, customCode)` - Creates test project
   - `RunDotNetCommand(dir, args)` - Runs dotnet CLI commands
3. Assert expectations
4. Run the test to verify

## CI/CD Integration

These tests should be run:
- Before publishing packages to NuGet
- On every PR that changes package structure
- As part of the release pipeline

## Contributing

When modifying package structure:
1. Update relevant tests
2. Run all tests locally
3. Fix any failures before committing
4. Update TEST_RESULTS.md if needed
